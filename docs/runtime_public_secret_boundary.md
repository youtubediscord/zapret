# Public Runtime Config And Secret Boundary

Дата фиксации: 2026-04-10

Этот документ объясняет границу между публичным runtime-конфигом GUI и настоящими секретами.

## Публичный runtime config

Файл `public_zapretgui/src/config/_build_secrets.py` в этом проекте называется так исторически, но по факту он используется как runtime-config слой для GUI.

Он может содержать только те значения, которые допустимо доставлять в клиентский runtime:

- `PREMIUM_API_BASE_URL`
- `UPDATE_SERVERS`
- `TG_UPDATE_BOT_TOKEN`, если это сознательно разрешённый runtime токен для private client build
- `PROXY_PRESETS`
- `MTPROXY_LINK`

То есть этот слой не является местом для настоящих приватных серверных секретов.

## Что должно оставаться пустым или вне GUI runtime

- серверные токены, не нужные клиенту во время обычного runtime;
- токены, которые дают чувствительный доступ к backend-операциям;
- build/deploy секреты;
- SSH-ключи, пароли, служебные deploy credentials.

Если поле не должно жить в клиенте, оно не должно “на всякий случай” попадать в `_build_secrets.py`.

## Правило границы

- Всё, что нужно GUI в runtime и допустимо ship'ить клиенту, должно жить в generated runtime config.
- Всё, что не должно ship'иться клиенту, должно оставаться в build/deploy контуре и не иметь runtime fallback в GUI.

## Почему это важно

- Если runtime начинает читать секреты из env/локальных файлов, появляется скрытый второй контур конфигурации.
- Если `_build_secrets.py` содержит лишние чувствительные данные, клиентская сборка перестаёт быть безопасной по границе ответственности.
- Поэтому правило простое: один явный runtime-config слой для публично допустимых значений и отдельный build/deploy слой для настоящих секретов.
